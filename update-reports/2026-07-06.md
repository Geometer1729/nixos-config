## SSH_AUTH_SOCK — delayed-failure risk to check after next reboot

The old (unstable) home-manager shipped a `set-SSH_AUTH_SOCK` user service that pushed `SSH_AUTH_SOCK` into the systemd/D-Bus user activation environment. home-manager release-26.05 replaced this with shell-init-only handling (`sshAuthSock.initialization` → zsh `envExtra` etc.), so that unit is now gone.

- Interactive shells (and anything they spawn) still get `SSH_AUTH_SOCK` — no change.
- systemd user units and dbus-activated GUI apps will no longer find it in the manager environment **after the next reboot/re-login**. Right now the live manager env still carries the old value (`systemctl --user show-environment` shows it), so nothing breaks until logout — a classic hidden-until-reboot change.
- Check after reboot: `systemctl --user show-environment | grep SSH_AUTH_SOCK`. If something launched outside a shell needs the agent, add `systemd.user.sessionVariables.SSH_AUTH_SOCK = "$XDG_RUNTIME_DIR/ssh-agent"` (environment.d) or an equivalent unit in `modules/home/ssh.nix`.
